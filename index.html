java面向对象

发表于 2017-06-18 | 分类于 javaSE

java面向对象

  • 面向对象特征

    • 封装
    • 继承
    • 多态

  • 类和对象的区别

    • 对象是类的具体实现, 类本身只是一种抽象, 一种规范;
    • 一个类可以有多个实例, 也就是一个类能够有多个对应的对象

成员变量和局部变量的区别

  • 类中的位置不同:
    • 成员变量: 在类中方法外
    • 局部变量: 在方法定义中或者方法声明上
  • 在内存中的位置不同
    • 成员变量: 在堆内存中
    • 局部变量: 在栈内存中
  • 生命周期不同:
    • 成员变量: 随着对象的创建而创建, 随对象的消失而消失
    • 局部变量: 随着方法的调用而存在, 随着方法的调用完毕而消失
  • 在部变量名称可以和成员变量名称一样, 在方法中使用的时候, 采用的是就近原则

基本类型和引用数据类型

  • 基本数据类型
    • char, byte, short, int, long, float, double, boolean
  • 引用数据类型
    • 数组, 类, 接口, 枚举

匿名对象

  • 没用名字的对象
    • 在仅仅使用一次的时候, 可以使用匿名对象, 可以减少代码!

封装

  • 概述
    • 隐藏对象属性和实现细节, 仅对外提供公共访问方式
  • 封装好处
    • 提高安全性
    • 提高代码复用性
  • 封装原则
    • 将不需要对外提供的内容都隐藏起来
    • 把属性隐藏起来, 提供公共的方法对其访问

构造方法 constructor

  • 概述
    • 给对象的数据(属性)进行初始化
  • 特点
    • 方法名与类名相同(大小也要与类名一致)
    • 没有返回值类型, 连 void 都没有
    • 没有具体的返回值 return

构造方法的重载

  • 重载: 方法名相同, 与返回值类型无关, 只看参数列表
  • 注意事项
    • 如果我们没有给出构造方法, 系统将自动提供一个无参构造方法
    • 如果我们给出了构造方法, 系统将不再提供默认的无参构造方法
    • 如果我们给出了构造方法, 但是还想要使用无惨的构造方法, 就必须自己给出, 建议永远自己给出无参构造

方法重写

  • 子类和父类都有一模一样的方法, 就是方法重写

  • 注意事项

    • 父类私有方法不能重写, 因为它根本不能继承下来
    • 子类重写父类方法时, 访问的权限不能更低
    • 父类的静态方法, 子类必须重写

    static 关键字

    • 随着类的加载而加载
    • 优于对象存在
    • 被类的所有对象共享
    • 可以通过类名直接调用
    • 注意事项
      • 静态方法中没有 this 关键字, 因为静态方法是随着类加载而加载, this 是随着对象的创建而创建
      • 静态方法只能访问静态的成员变量和静态的成员方法,即静态只能访问静态

静态变量和成员变量的区别

  • 静态变量也叫类变量, 成员变量也叫对象变量
  • 所属不同
    • 静态变量属于类, 也叫类变量
    • 成员变量属于对象, 所以也称之为实例变量
  • 内存中存储位置不同
    • 静态变量存储于方法区的静态区
    • 成员变量存储于堆内存
  • 内存初选时间不同
    • 静态变量随着类的加载而加载, 随着类的消失而消失
    • 成员变量随着对象的创建而创建, 随着对象的消失而消失
  • 调用不同
    • 静态变量可以通过类名调用,也可以通过对象调用
    • 成员变量只能通过对象调用
  • 制作工具类, 使用静态方法, 不用 new 直接调用

main 方法解释

  • public: 被 JVM 调用, 访问权限足够大
  • static: 被 JVM 调用, 不用创建对象
  • void: 不需要给 JVM 返回值
  • main: 一个通用的,名称, 不是关键字, 但是被 JVM 识别, JVM 默认会加载 main 方法
  • String[] args: 用于键盘录入

代码块

  • {} 称之为代码块
  • 分类
    • 局部代码块: 在方法中出现, 限定变量的生命周期, 及早释放内存, 提高内存利用率
    • 构造代码块: 在类方法外出现, 多个构造方法中相同的代码存在到一起, 每次调用构造都执行, 并且在构造方法前执行
    • 静态代码块: 在类方法外出现, 并加上 static 修饰, 用于给类进行初始化, 在加载时候就执行, 并且只执行一次, 一般用于加载驱动

继承

  • 继承好处
    • 提高代码复用
    • 提高代码维护
    • 让类与类之间产生了关系, 是多态的前提
  • 继承弊端
    • 类的耦合性增强了
  • 注意事项
    • 子类只能继承父类所有的非私有的成员
    • 子类不能继承父类的构造方法, 但是可以通过 super (当前对象父类的引用) 来访问父类的构造方法,或者 super. 来访问父类的成员
  • Java 只支持单继承, 不支持多继承

final 修饰符

  • 特点
    • 修饰类: 类不能继承
    • 修饰变量: 变量不再次赋值, 变成常量
    • 修饰方法, 方法不能被重写
  • 修饰基本类型,值不能变, 修饰引用类型, 地址不能点, 但是对象中的属性是可以改变的

内部类

  • 特点
    • 内部类可以直接访问外部的成员, 包括私有
    • 外部类要访问内部类的成员, 必须创建对象
    • new外部类().new内部类().属性

== 和 equals 的区别

  • == 是一个比较运算符, 既可以比较基本数据类型, 也可以比较应用数据类型, 基本能数据类型比较值, 引用数据类型比较的是地址值
  • equals 方法, 只能用来比较引用数据类型, 所有的对象都会继承 Object 类中的方法, 如果没有重写, equals 和 == 比较引用数据类型无区别, 重写
    后的 equals 比较的是对象中的属性

React

发表于 2017-06-12 | 分类于 React

React 特点

  1. 简单
  2. 声明式
    React 的核心是组件, 组件的设计目的是提高代码复用率, 降低代码的测试难道, 降低代码复杂度;

jsx 基础语法

  1. 首字母大写代表 React 组件, 小写代表是 DOM 元素;
  2. React 组件支持嵌套;
  3. {} 里面可以写 javascript 求值表达式代码 if 判断需要使用 三元表达式或者短路运算符;
  4. 组件驼峰命名;
  5. html 和 class 是 javascript 的关键字, 需要使用 htmlFor 和 className 代替;
  6. HTML 中写字符串 {" "} 或者 不用 {}, 直接写;
  7. style = {}, 必须是一个对象, 而且只能写在 HTML 的标签中, 写在标签对应得组件是无效的,只能当属性传递下去!;

非 DOM 属性

  1. dangerouslySetInnerHTML: 在 JSX 中直接插入 HTML, 用于动态 DOM 元素, 可能导致 XSS 攻击;
  2. ref: 父组件引用子组件;
    在节点中设置 ref=”自定义”, 然后在其他位置或方法可以获取该节点的 DOM 实例节点, this.refs.”自定义”.属性;
  3. key: 提高渲染性能,
    React 并不能知道某个节点在位置不同的情况下,在组件中是否能复用, 导致插入节点使得可复用的节点无法复用, 引入 key 可以复用这个节点, 因为React 会比较 key 是否相同, 如果相同,则复用改节点, 直接在前插入一个新节点, 在列表中,应该使用 key 以提高渲染性能;

React diff 算法流程

  1. 比较节点是否相同, 相同进入下一步, 不相同则直接生成新的节点;
  2. 判断是否是 DOM 还是 自定义的节点, 进入下一步;
  3. 比较节点的属性及ID, class, 相同结束比较,如果不同, 这重新设置这些属性不会重新渲染,如果是组件的话,重新设置状态;

JSX 解释器架构介绍

  1. 入口函数 -> 加载模块 -> 解析 JSX -> 执行 JS
    因为所有的程序运行在浏览器中, 大量的 JSX 会有性能问题, 解决办法是,提前编译 JSX

React 组件的生命周期

  1. 组件本质上是状态机;
  2. 钩子函数
    1. 初始化
    • getDefaultProps: 第一个实例创建时调用,只调用一次, 实例之间共享引用;
    • getInitialState: 初始化每个实例特有的状态;
      以上两个在ES6 中用 constructor 执行
    • componentWillMount: 组件开始挂载 DOM 上, render 之前最后一次修改状态的机会;
    • render: 渲染函数, 只能访问 this.props 和 this.state, 只有一个顶层组件, 不允许修改状态和 DOM 输出;
    • componetDidMount: 成功渲染并完成真实 DOM 之后触发, 所有的 DOM 操作, 都应该在这里使用, 因为,在此之前, DOM 并没有渲染完成!
    1. 运行中
    • componentWillReceiveProps: 在父组件修改属性触发, 在传送新的属性到达组件之前;
    • shouldComponentUpdate: 返回 false 会阻止 render 函数运行, 避免不需要的 diff 用于提高性能;
    • componentWillUpdate: 不能修改属性和状态;
    • render: 渲染函数, 只能访问 this.props 和 this.state, 只有一个顶层组件, 不允许修改状态和 DOM 输出;
    • componetDidUpdate: 可以修改 DOM;
    1. 销毁
    • componentWillUnmont: 组件销毁时触发;

属性和状态

  1. porps 可以接受由父组件中传来的属性;

  2. state 状态管理对象,当对象中的任意属性改变, 那么他会调用 render 方法, 重新渲染!
    重新设置状态只能用 setState() 方法来修改区别
    当运行时需要改变的属性, 必须使用过 state, 不能使用 props;
    props 是自顶向下流动的, 组件不能改变自身的 porps;

事件处理

`onClick={this.handleClick.bind(this)}`
on后面的事件名字必须大写;
`{}`, 里面的方法不用加()调用;
bind(this), 将改组件的对象传入, 这里就能在方法里用 this 来获取到改组件的属性: 例如 this.refs;
调用方法, 默认会传入一个事件对象, 可以在该方法中获取;bind(this, value1), 后面可以传入一个值,这样可以在该调用的函数中获取, (value1, e), 事件参数,默认是最后一位的, 前面的参数才是传递过来的参数;
子组件的参数传递给父组件, 是父组件把自己的函数当做属性传递给子组件, 然后子组件在自己的事件中, 调用父组件传递过来的函数, 在该函数中,可以拿到事件对象 e, e.target, 就是子组件;

可控组件和不可控组件

  1. 区别: 可控组件由 React 管理, 不可控组件是直接获取 DOM, 不需经过 React

    可控的好处

    1. 符合 React 的数据流
    2. 数据存储在 state 中, 便于使用,效率更高
    3. 便于对数据进行处理

Flux

  • Flux 不是具体的框架, 而是 Facebook 提出的一种代码架构
  • React 只是一个视图库, Flux 是在 React 基础上对于前端整体的组织方案
  • 传统的 MVC 只适用于中小型系统, 对于大规模系统来说 MVC 复杂度过高
  • Flux 的目的是保证逻辑清晰, 数据流向清晰, 依赖关系清晰

immutable 笔记

发表于 2017-01-08 | 分类于 React

开始

用 npm 安装 immutable 。

npm install immutable

然后在任何模块中引入

var Immutable = require(‘immutable’);
var map1 = Immutable.Map({a:1, b:2, c:3});
var map2 = map1.set(‘b’, 50);
map1.get(‘b’); // 2
map2.get(‘b’); // 50

浏览器

在浏览器中使用 immutable, 下载 dist/immutable.min.js 或者使用 CDNJS 或 jsDelivr.

然后在 script 中引入

&lt;script src="immutable.min.js"&gt;</script>
<script>
var map1 = Immutable.Map({a:1, b:2, c:3});
var map2 = map1.set(‘b’, 50);
map1.get(‘b’); // 2
map2.get(‘b’); // 50
</script>

AMD 加载器 (RequireJS):

require([‘./immutable.min.js’], function (Immutable) {
var map1 = Immutable.Map({a:1, b:2, c:3});
var map2 = map1.set(‘b’, 50);
map1.get(‘b’); // 2
map2.get(‘b’); // 50
});

如果使用 browserify，immutable 在浏览器同样适用。
TypeScript

在 TypeScript 中使用 Immutable 的不变集合和序列犹如原生集合一样，获得泛型，异常检查，自动完成的等 IDE 功能。 只需要在文件顶部添加相对路径的引用即可。

///<reference path=’./node_modules/immutable/dist/immutable.d.ts’/>
import Immutable = require(‘immutable’);
var map1: Immutable.Map<string, number>;
map1 = Immutable.Map({a:1, b:2, c:3});
var map2 = map1.set(‘b’, 50);
map1.get(‘b’); // 2
map2.get(‘b’); // 50

不变性再思

跟踪变化和维护状态通常是的应用开发困难。用不可变数据开发应用引导你用不同方式思考数据流。

在应用中订阅数据事件，导致大量记账开销会损害性能，甚至很严重，并使得应用丢失一致性导致错误孕育温床。如果不应该改变的数据不会变化，那么在模型中监听变化是由上而下单向传递。

这种数据流模型和 React 的架构是一致的， 并且和 Flux 高度契合。

如果数据只是传递而不是被监听，便只需要关心当数据改变了以后做什么事，你可以使用等式。

不变集合应当按 值 来对待， 而不是 对象 。因为对象会随着时间变化，值代表着某一时间的状态。这个原则对理解和正确使用不变数据非常重要。 其中关键在于使用 Immutable.is() 或者 .equals() 函数来判断值相等而不是 === 操作符。

var map1 = Immutable.Map({a:1, b:2, c:3});
var map2 = map1.set(‘b’, 2);
assert(map1.equals(map2) === true);
var map3 = map1.set(‘b’, 50);
assert(map1.equals(map3) === false);

注：为了优化性能， Immutable 在判断不变操作后会尝试返回原集合，使得 === 操作符能正确判断是否值相等。比如这在记忆函数内部，如果深度判断参数是否改变而需要重新计算，花费可能比计算本身更加耗时， === 操作符在 Immutable.is 和 .equals() 内部同样被使用以优化性能。

如果对象不可变，可以通过引用复制来进行克隆。因为引用比对象开销更小，可以节省内存和提升依赖复制的程序的执行速度（比如撤销队列 undo-stack）。

var map1 = Immutable.Map({a:1, b:2, c:3});
var clone = map1;

JavaScript 风格的 API

immutable 灵感启发来自 Clojure, Scala, Haskell 和其它函数式编程环境，并专门为 JavaScript 设计，所以仿照 ES6 Array, Map, 和 Set 设计了一套面向对象的接口。

区别在于，通常会改变集合的方法如 push, set, unshift 或者 splice ，不变集合只会返回一个新的不变集合。至于返回新数组的方法如 slice 或 concat ，不变集合同样返回新的不变集合。

var list1 = Immutable.List.of(1, 2);
var list2 = list1.push(3, 4, 5);
var list3 = list2.unshift(0);
var list4 = list1.concat(list2, list3);
assert(list1.size === 2);
assert(list2.size === 5);
assert(list3.size === 6);
assert(list4.size === 13);
assert(list4.get(0) === 1);

几乎所有 Array 的方法都可以在 Immutable.List 找到等价方法，Map 对应 Immutable.Map, 而 Set 对应于 Immutable.Set, 包括 forEach() 和 map() 的集合操作。

var alpha = Immutable.Map({a:1, b:2, c:3, d:4});
alpha.map((v, k) => k.toUpperCase()).join();
// ‘A,B,C,D’

接受原生 JavaScript 对象

设计支持现有 JavaScript 代码， immutable 所有 Iterable 参数接收原生 JavaScript 数组和对象并性能无损。

var map1 = Immutable.Map({a:1, b:2, c:3, d:4});
var map2 = Immutable.Map({c:10, a:20, t:30});
var obj = {d:100, o:200, g:300};
var map3 = map1.merge(map2, obj);
// Map { a: 20, b: 2, c: 10, d: 100, t: 30, o: 200, g: 300 }

之所以这样是因为 immutable 将 JavaScript Array 或 Object 以 Iterable 处理。可以利用这点在 JavaScript 对象上完成复杂的集合方法，否则原生 API 比较松散。因为 Seq 惰性执行并且不创建中间结果，操作非常高效。

var myObject = {a:1,b:2,c:3};
Immutable.Seq(myObject).map(x => x * x).toObject();
// { a: 1, b: 4, c: 9 }

需要强调的是，使用 JS 对象构建 Immutable Maps，对象的属性可以用 去-括号 简写方式访问，然而 Immutable Maps 的类型是不定的。

var obj = { 1: “one” };
Object.keys(obj); // [ “1” ]
obj[“1”]; // “one”
obj[1]; // “one”

var map = Immutable.fromJS(obj);
map.get(“1”); // “one”
map.get(1); // undefined

JavaScript 对象的属性访问会被转换成字符串，但是 Immutable Maps 索引支持任何类型，get() 函数的参数不会被改变。
转换成原生 JavaScript 对象

所有 immutable Iterables 可以转换成原生 JavaScript Array 和 Object，使用 toArray() 和 toObject() 是浅拷贝式的，或者使用深拷贝式的 toJS()。所有 Immutable Iterable 实现了 toJSON 行为以兼容 JSON.stringify.

var deep = Immutable.Map({ a: 1, b: 2, c: Immutable.List.of(3, 4, 5) });
deep.toObject() // { a: 1, b: 2, c: List [ 3, 4, 5 ] }
deep.toArray() // [ 1, 2, List [ 3, 4, 5 ] ]
deep.toJS() // { a: 1, b: 2, c: [ 3, 4, 5 ] }
JSON.stringify(deep) // ‘{“a”:1,”b”:2,”c”:[3,4,5]}’

拥抱 ES6

Immutable 支持 ES6 最新版本的特性，包括 Iterators, Arrow Functions, Classes, 和 Modules. 它受到 ES6 Map 和 Set 集合的启发。类库编译成 ES3 以兼容所有主流浏览器。

所有的代码用 ES6 呈现。要想兼容所有浏览器，需要转换成 ES3.

// ES6
foo.map(x => x x);
// ES3
foo.map(function (x) { return x x; });

嵌套结构

immutable 集合支持深度嵌套，类似JSON

var nested = Immutable.fromJS({a:{b:{c:[3,4,5]}}});
// Map { a: Map { b: Map { c: List [ 3, 4, 5 ] } } }

一些有用的支持对嵌套数据读写的函数。常用的有 mergeDeep, getIn, setIn, updateIn, 在 List, Map 和 OrderedMap .

var nested2 = nested.mergeDeep({a:{b:{d:6}}});
// Map { a: Map { b: Map { c: List [ 3, 4, 5 ], d: 6 } } }

nested2.getIn([‘a’, ‘b’, ‘d’]); // 6

var nested3 = nested2.updateIn([‘a’, ‘b’, ‘d’], value => value + 1);
// Map { a: Map { b: Map { c: List [ 3, 4, 5 ], d: 7 } } }

var nested4 = nested3.updateIn([‘a’, ‘b’, ‘c’], list => list.push(6));
// Map { a: Map { b: Map { c: List [ 3, 4, 5, 6 ], d: 7 } } }

惰性 Seq

Seq 描述了延迟操作，高效的串连 Iterable 方法（比如 map 和 filter ）。

Seq 是不可变的 — 一旦 Seq 创建便不能修改, 添加, 排列或其它修改. 任何修改操作都会生成一个新的 Sep.

Seq 是惰性的 — Seq 尽可能少的去相应函数调用。

比如，下面代码不会发生任何操作，因为结果没有被用到：

var oddSquares = Immutable.Seq.of(1,2,3,4,5,6,7,8)
.filter(x => x % 2).map(x => x * x);

一旦 Seq 被使用，它只会执行必要的工作。在这个例子中，没有中间数组被创建出来，filter 被调用了三次， map只被调用了两次：

console.log(oddSquares.get(1)); // 9

所有集合可以用 .toSeq() 转换成 Sep.

var seq = Immutable.Map({a:1, b:1, c:1}).toSeq();

Seq 允许高效串联操作，尤其当需要转换类型的时候（比如转换成 JS 对象）：

seq.flip().map(key => key.toUpperCase()).flip().toObject();
// Map { A: 1, B: 1, C: 1 }

还可以表达正常情况会导致内存溢出的逻辑：

Immutable.Range(1, Infinity)
.skip(1000)
.map(n => -n)
.filter(n => n % 2 === 0)
.take(2)
.reduce((r, n) => r * n, 1);
// 1006008

注： iterable 的迭代顺序是固定的，然而顺序可能不明确，比如 Map。
集合等价于数据

Immutable 等号判断把不变数据结构按照存数据处理，在必要的时候进行深度等号判断。

var map1 = Immutable.Map({a:1, b:1, c:1});
var map2 = Immutable.Map({a:1, b:1, c:1});
assert(map1 !== map2); // two different instances
assert(Immutable.is(map1, map2)); // have equivalent values
assert(map1.equals(map2)); // alternatively use the equals method

Immutable.is() 的实现基于 Object.is 上判断都是 immutable 类型且所有的 key 和 value 都相等。
批量更改

If a tree falls in the woods, does it make a sound? > > If a pure function mutates some local data in order to produce an immutable > return value, is that ok? > > — Rich Hickey, Clojure

每次修改生成一个新的 immutable 对象有一定性能损耗。如果需要在返回结构之前做一连串的修改， Immutable 提供 withMutations 方法，使批量修复在临时集合拷贝上发生。实际上 Immutable 内部也是这样处理复杂改动的。

举例说明，通过一次创建完成 list2 而不是3次。

var list1 = Immutable.List.of(1,2,3);
var list2 = list1.withMutations(function (list) {
list.push(4).push(5).push(6);
});
assert(list1.size === 3);
assert(list2.size === 6);

注： immutable 还提供了 asMutable 和 asImmutable ， 但是只有在 withMutation 不能满足情况的时候使用。注意不要返回一个可变的拷贝，会导致非预知的行为。

重要！：只有特殊的几个方法能使用 withMutation 其中包括 set, push 和 pop。这些方法能直接作用于内部数据结构，而方法如 map, filter, sort, splice 一定不会修改可修改集合并返回新的不变集合。

集合的由来 Collection

  • 数组的长度是固定的, 当添加的元素超过了数组的长度时需要重新定义, 过于麻烦, java 内部提供了集合lei,能存储任意对象, 长度时可变的, 随着元素的增加而增加, 随着元素的减少而减少

集合与数组的区别

  • 数组即可以存储基本数据类型, 也可以存储引用数据类型, 基本数据类型存储的是值, 应用类型存储的是地址, 但是集合只能存储引用数据类型(对象), 集合中也可以存储基本数据但是会自动装箱成引用数据类型(int->integer)
  • 数组长度固定, 集合长度是可变的

数据结构之数组和链表

  • 数组查询快, 修改快, 增加删除慢
  • 链表查询慢, 修改慢, 增加删除快

HashSet (底层hash算法)

  • 原理
  • 我们使用 set 集合都是需要去掉重复的元素, 如果在存储的时候逐个 equals() 比较, 效率太低, 哈希算法提高了去重复的效率, 降低了使用 equals() 方法的次数
  • 如果没有哈希值相同的对象就存入集合, 如果有逐个进行 equals() 比较, 结果为 false 就存入, true 则不存
  • 底层比较的是对象的地址, 如果要比较值, 需要重写内部的 hashCode() 和 equals()
  • 将自定义的对象存入 HashSet 去chongf,必须重写 hashCode() 和 equals() 方法
  • hashCode() 属性相同的对象返回值必须相同, 属性不相同的返回值尽量不同(提高效率)
  • equals() 属性相同返回 true, 属性不同返回 false, 返回 false 时存储

LinkedHashSet 可以保证存取的顺序

TreeSet(二叉树算法)

原理

  • 特点
  • TreeSet 是用来排序的, 可以指定一个顺序, 对象存入后, 会按照指定的顺序排序
  • 使用方法
    • 自然排序(Comparable)
      • TreeSet 类的add()方法中会把存入的对象提升为 Comparable 类型
      • 调用 compareTo() 方法和集合中的对象比较
      • 根据 compareTo() 方法返回的结果进行存储
    • 比较器顺序(Comparator)
      • 创建 TreeSet 的时候可以制定一个 Comparator
      • 如果传入了 Comparator 的子类对象, 那么 TreeSet 就会按照比较器中的顺序排序
    • 两种方式的区别
      • TreeSet 构造函数什么都不传, 默认按照类中 Comparable 的顺序
      • TreeSet 如果传入 Comparator, 就优先按照 Comparator
